# Demo 案例规范

本文件服务于后续 agent 和维护者统一整理站内 Demo，不属于站点正文内容。正式页面仍优先保持“问题 / 结论 / Demo / 参考来源”的读者视角。

## 目标

- Demo 用来验证或辅助理解正文结论，不替代结论中的知识框架。
- Demo 应让读者能看出“页面效果由哪段 HTML/CSS 造成”。
- Demo 的复杂度应服务题目本身，避免把教学案例做成新的理解负担。

## 内容选择

- 优先把远程 CodePen、CodeSandbox、不可控 iframe 改成本地组件。
- 保留能验证核心规则的最小案例；删除只展示结果、无法说明规则的装饰性案例。
- 多方案案例要先判断读者是否需要同时比较：
  - 需要横向比较差异时，优先使用竖向排列、网格排列或对比表。
  - 每次只需要理解一个方案，且案例会明显拉长页面时，可以使用 tab、分段按钮或折叠切换。
  - 不强制所有多方案案例都使用 tab。

## 布局规则

- 常规交互案例优先采用左侧效果、右侧代码的布局。
- 小屏幕下允许上下排列，但仍要保持效果区和代码区相邻。
- 案例区域应设置合理最大高度；内容过长时提供展开能力，避免单个 Demo 把页面拉得过长。
- 不把卡片嵌套成多层卡片；重复案例可以用轻量边框或分隔线区分。

## 代码一致性

- 右侧代码必须和左侧真实实现一致，不能出现旧类名、旧尺寸、伪代码或未使用属性。
- 如果代码为教学裁剪版，必须保证裁剪后仍能解释左侧效果的关键机制。
- 修改 Demo 后检查每个方案的 HTML 结构、CSS selector、尺寸变量和实际渲染是否同步。

## 教学标识

- 当代码和渲染元素难以建立联系时，可以在教学 Demo 中添加辅助标识：
  - selector chip：展示当前案例的关键选择器。
  - `data-css`：标记关键 DOM 元素对应的 CSS selector。
  - 简短标题：区分每个方案或状态。
- 这些标识只用于文档教学和调试，不作为真实业务代码的语义建模建议。
- 标识文字应短且不遮挡布局；必要时使用截断或 tooltip。

## 交互与可访问性

- 可切换 Demo 应支持键盘操作和清晰的可访问名称。
- 展开、折叠、tab、分段按钮等控件应有明确的当前状态。
- Demo 不应依赖 hover 才能理解核心内容。

## 验证

- 修改 Vue/VitePress 组件后运行：

```bash
npm run typecheck
npm run build
```

- 视觉或交互变化较大时，用浏览器检查：
  - 每个方案切换后左侧效果和右侧代码是否同步。
  - 是否存在横向溢出、文字遮挡、内容被压缩到不可读。
  - 默认高度和展开状态是否符合阅读体验。
