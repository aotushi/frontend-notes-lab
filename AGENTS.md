## 文档更新策略

- 正文优先保持面试题密度，常规结构为：问题、结论、Demo、面试回答、参考来源。
- 结论要先给可直接背诵和判断的答案，再补必要解释；不要把长篇推导默认铺在正文中。
- 难理解的结论使用 `ConceptNote` 做渐进解释：正文保留原目标句，并把该句作为可点击触发器。
- `ConceptNote` 的 `label` 应是正文里的原结论句或术语短句；解释内容放在组件插槽中。
- 桌面端解释采用右侧面板；移动端采用底部抽屉。不要为单篇文档重复实现独立弹窗。

## 权威链接策略

- 结论中的校准链接优先指向 MDN、WHATWG、W3C、web.dev、Chrome Developers 等权威资料。
- 链接要尽量定位到具体小节、锚点或 text fragment；不要只链接到资料首页或元素页面顶部，除非没有更精确位置。
- 站内链接用于串联相关知识点；权威外链用于证明定义、规范行为和浏览器平台事实。
- 参考来源列表可以放页面级资料入口，但正文中的争议点或关键定义应尽量链接到具体位置。

## 可视化与释义策略

- 图示必须服务于理解，不做装饰。
- 优先使用 HTML/CSS 盒模型、状态块、对比表、流程列表等可维护结构。
- 只有当 SVG 能表达用户能直接察觉的关系时才使用 SVG；装饰性箭头或意义不清的图形应删除。
- Demo 应能验证结论；释义图示用于帮助理解，不替代可验证 demo。

## 时序图设计规范

- 加载、解析、执行、事件触发类时序图必须优先使用 `TimelineTrace` 组件。
- 具体规范见 `apps/docs/docs/agents/timeline-trace-guidelines.md`。
- Markdown 文档中不允许手写一次性 `<svg>` 时序图。
- Markdown 文档中不允许手写 `.timeline__*` / `.trace-map` / `.connector-*` 这类页面私有时序图样式。
- 横条表示持续过程，短竖条表示瞬时事件，虚线表示等待或依赖关系。
- 颜色只由 `type` 决定：`html` 蓝色、`css` 绿色、`js` 紫色、`event` 红色、`resource` 橙色。
- 不使用数字圆点表示事件顺序；事件顺序由 X 轴位置表达。
- 概念图和实测日志应分离：概念图展示稳定规则，实测日志展示本次浏览器证据。
- 涉及时序图新增或修改时运行 `npm run lint:timeline`。

## 候选题与迁移审计

- 迁移数量、候选题处理、审计过程、覆盖情况只在聊天中说明。
- 不把“迁移审计”“原答案审计”“覆盖候选题”等过程性内容写入站点文档。
- 从候选题库迁移内容时，先判断质量、时效性、分类归属，再合并到目标主题。
- 过时或低质量题目可以删除，但删除原因应在聊天审计中说明，不写入目标页面。

## 组件维护

- VitePress 自定义组件放在 `apps/docs/docs/.vitepress/theme/components/`。
- 新增或修改 Vue 组件时使用 Vue 3 Composition API 和 `<script setup lang="ts">`。
- 公共组件优先复用和扩展，避免为单个页面复制一套交互。
- 自定义组件应支持键盘操作和可访问名称；弹层至少支持关闭按钮、遮罩关闭和 Esc 关闭。

## 验证

- 修改 Markdown 内容后，按影响范围决定是否运行构建；涉及组件、配置或全站导航时必须运行验证。
- 修改 Vue/VitePress 组件后运行：

```bash
npm run typecheck
npm run build
```

- 构建中的既有 chunk size warning 可以保留，但新的类型错误、渲染错误或文档解析错误必须修复。
