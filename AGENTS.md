## 本地写作 Skills

- 项目级 skills 安装在 `.agents/skills/`，用于提升资料收集、翻译和 Markdown 输出质量。
- 整理文章、面试题草稿或长 Markdown 时，优先考虑 `baoyu-format-markdown`，但仍遵守本站“问题、结论、Demo、参考来源”的页面结构。
- 从网页沉淀参考资料时，优先考虑 `baoyu-url-to-markdown`；抓取后必须再做权威性、时效性、版权和是否适合面试题的人工判断。
- 翻译英文资料或规范说明时，优先考虑 `baoyu-translate`；翻译结果不能直接当作结论，仍需用 MDN、WHATWG、W3C、web.dev、Chrome Developers 等来源校准。
- 宝玉 skills 生成的分析文件、中间稿、发布稿不应默认进入站点文档；只有经过审计后的正文内容才写入 `apps/docs/docs/`。

## 面试题更新进度

- 继续整理或迁移面试题前，先读取 `.agents/interview-update-progress.md`。
- 该文件记录当前推进模块、进度锚点、统计口径和下一步范围；它服务于 agent 续接工作，不属于站点正文。
- `.scratch/` 只作为候选题语料、外部题库缓存、日志和截图等临时工作区，不作为唯一进度来源。

## 文档更新策略

- 正文优先保持面试题密度，常规结构为：问题、结论、Demo、参考来源。
- `结论` 是完整答案区，可以根据题型包含定义、分类、规则、应用、误区、对比表或流程；不要再单独添加“面试回答”模块。
- 结论要直接回答问题，避免“面试中应该”“需要抓住”等过程性表达。
- 概述型概念题不能只写功能结论；应保留概念来源、所属分类、核心定义、关键规则和典型应用，避免把知识点压缩成技巧列表。
- 概述型或机制型问题优先补充“理解路径”：说明本题依赖哪些前置概念，以及这些概念如何递进到当前结论。理解路径要短，只保留真正影响理解的依赖关系。
- 机制型、概念型问题优先使用“定义 -> 组成 -> 规则 -> 图示 -> 边界”的概念递进结构；不要为了增加 `###` 问题数量，把一个完整认知单元拆成多个并列问题。
- 从 raw-data 迁移机制型、概念型内容时，先识别原文的认知顺序；如果原内容已经是可复用的概念递进结构，应直接沿用该结构，只修正事实、术语、私有语法和过时表达。
- 修正旧答案时，先识别原答案的认知路径和覆盖角度；除非结构本身错误，否则优先保留结构，再修正事实、术语和边界。
- 准确性不能以牺牲可理解性为代价；把绝对表述改成边界表述时，应保留原句中有价值的直观模型。
- Demo 只负责验证或辅助理解，不替代结论中的知识框架。
- `ConceptNote` 只包住需要解释的术语、短语或关键判断，不包整句，除非整句本身就是不可拆的结论。
- 正文结构不能因为插入 `ConceptNote` 被破坏；列表仍按列表展示，段落仍按段落展示。
- 弹窗内容应结构化，优先包含：直观理解、示例、边界。避免只有一段长文本。
- 遇到“部分、通常、可能、隔离、上下文、方向”等限定词时，优先解释适用范围和不适用范围。
- 在 Markdown 列表中使用复杂弹窗内容时，优先使用 `sections` 结构化属性，避免在列表项里写跨段落 slot。
- 难理解的结论使用 `ConceptNote` 做渐进解释：正文保留原目标句，并把该句作为可点击触发器。
- `ConceptNote` 的 `label` 应是正文里的原结论句或术语短句；解释内容放在组件插槽中。
- 桌面端解释采用右侧面板；移动端采用底部抽屉。不要为单篇文档重复实现独立弹窗。

## 权威链接策略

- 结论中的校准链接优先指向 MDN、WHATWG、W3C、web.dev、Chrome Developers 等权威资料。
- 链接要尽量定位到具体小节、锚点或 text fragment；不要只链接到资料首页或元素页面顶部，除非没有更精确位置。
- 站内链接用于串联相关知识点；权威外链用于证明定义、规范行为和浏览器平台事实。
- 参考来源列表可以放页面级资料入口，但正文中的争议点或关键定义应尽量链接到具体位置。

## 可视化与释义策略

- 图示必须服务于理解，不做装饰。
- 优先使用 HTML/CSS 盒模型、状态块、对比表、流程列表等可维护结构。
- 只有当 SVG 能表达用户能直接察觉的关系时才使用 SVG；装饰性箭头或意义不清的图形应删除。
- Demo 应能验证结论；释义图示用于帮助理解，不替代可验证 demo。
- Demo 优先使用本地组件，不依赖远程 CodePen、CodeSandbox 或不可控 iframe。
- Demo 的展示方式由内容决定：需要同时比较时可竖向排列或网格排列；需要聚焦单个方案、控制页面高度时可使用 tab 或分段切换，不把某一种形式作为强制默认。
- Demo 左侧效果、右侧代码和真实实现必须一致；教学型案例可以用 selector chip、`data-css` 等辅助标识建立“元素 -> CSS -> 效果”的映射。
- 复杂 Demo 规范见 `.agents/demo-case-guidelines.md`。

## 时序图设计规范

- 加载、解析、执行、事件触发类时序图必须优先使用 `TimelineTrace` 组件。
- 具体规范见 `apps/docs/docs/agents/timeline-trace-guidelines.md`。
- Markdown 文档中不允许手写一次性 `<svg>` 时序图。
- Markdown 文档中不允许手写 `.timeline__*` / `.trace-map` / `.connector-*` 这类页面私有时序图样式。
- 横条表示持续过程，短竖条表示瞬时事件，虚线表示等待或依赖关系。
- 颜色只由 `type` 决定：`html` 蓝色、`css` 绿色、`js` 紫色、`event` 红色、`resource` 橙色。
- 不使用数字圆点表示事件顺序；事件顺序由 X 轴位置表达。
- 概念图和实测日志应分离：概念图展示稳定规则，实测日志展示本次浏览器证据。
- 涉及时序图新增或修改时运行 `npm run lint:timeline`。

## 候选题与迁移审计

- 迁移数量、候选题处理、审计过程、覆盖情况只在聊天中说明。
- 不把“迁移审计”“原答案审计”“覆盖候选题”等过程性内容写入站点文档。
- 不把 `迁移自 raw-notes/...`、`以下内容先按原文完整迁移` 这类迁移来源或临时处理说明写入站点文档。
- 从 Obsidian 或其它个人笔记迁移时，必须移除 `[[...]]` 内链、孤立块引用、未解析附件语法等站点无法正确表达的私有笔记语法；需要保留引用关系时改成 VitePress 站内链接或权威外链。
- 从候选题库迁移内容时，先判断质量、时效性、分类归属，再合并到目标主题。
- 遇到“列举 N 种方式”“有哪些实现”“手写多种方案”这类多方案题，不得为了精简只保留推荐方案；应保留候选题中的主要方法清单，并为每种有效方法提供可运行实现或清晰伪代码。
- raw-data 中的明显错误应在正式正文里修正，而不是原样搬运；修正后不要写“raw-data 中写错了”“这里已修正”等过程性文字。
- 低质量或不推荐的方案可以保留，但点评必须聚焦方法质量，例如时间复杂度、空间开销、是否保序、是否修改原数据、类型边界、兼容性和工程可用性。
- 如果确实需要压缩、合并或跳过 raw-data 中的某些方案，必须保证不改变题目本身的考察意图；处理原因只在聊天审计或报告中说明，不写入正式页面。
- 过时或低质量题目可以删除，但删除原因应在聊天审计中说明，不写入目标页面。

## 组件维护

- VitePress 自定义组件放在 `apps/docs/docs/.vitepress/theme/components/`。
- 新增或修改 Vue 组件时使用 Vue 3 Composition API 和 `<script setup lang="ts">`。
- 公共组件优先复用和扩展，避免为单个页面复制一套交互。
- 自定义组件应支持键盘操作和可访问名称；弹层至少支持关闭按钮、遮罩关闭和 Esc 关闭。

## 部署策略

- Cloudflare Worker 已绑定 GitHub 仓库，`main` 分支推送后应由 Cloudflare 自动构建和部署。
- 日常发布优先使用：本地验证、提交、推送；不要把手动 `wrangler deploy` 当作默认发布步骤。
- 手动部署只用于排查自动部署失败、验证 Wrangler 配置或用户明确要求立即发布。
- 根目录 `npm run deploy` 仅执行 Worker 发布；需要本地先构建再发布时使用 `npm run deploy:full`。

## 验证

- 修改 Markdown 内容后，按影响范围决定是否运行构建；涉及组件、配置或全站导航时必须运行验证。
- 修改 Vue/VitePress 组件后运行：

```bash
npm run typecheck
npm run build
```

- 构建中的既有 chunk size warning 可以保留，但新的类型错误、渲染错误或文档解析错误必须修复。
